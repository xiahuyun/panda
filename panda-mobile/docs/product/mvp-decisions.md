# Panda Mobile MVP 决策记录 (2026-03-11)

## 1. 决策结论（已确认）

1. 需要将 `ui-schema` 收敛到 MVP 范围。
2. 路由结构可以拆分为真实页面路由，不再仅用 `pages/index/index` 演示页。
3. mock 验证码规则：
   - 验证码长度必须为 6 位
   - mock 测试码固定为 `123456`
4. MVP 登录失败仅处理一种错误：`验证码错误`，弹窗提示后 **3 秒自动关闭**。
5. MVP 不使用登录鉴权 token，不持久化登录状态；登录成功信息只保存在内存。
6. 缘分墙活跃态数据字段需要补全（见下文）。
7. UI 状态规则放宽：不强制所有页面都覆盖 `loading/empty/error/content` 四态，按页面类型定义。
8. 输入不合法时前端拦截并禁用提交，不触发 mock 校验。
9. 隐私政策/用户协议点击行为在 MVP 阶段先用 toast 占位，且文案区分。
10. 需要埋点方案，先本地打印事件，不接远端。
11. 缘分墙本轮仅做成功路径，不做错误分支。
12. 缘分墙 `membership_entry` 灰态展示，不可点。
13. `login_submit_blocked` 仅在输入状态从“合法 -> 非法”时触发一次。
14. 输入保持非法态期间不重复触发 `login_submit_blocked`，点击禁用按钮也不触发。
15. 错误弹窗 3 秒期间全屏完全禁交互（含输入、返回与任意页面点击）。
16. 协议/隐私 toast 触发需节流，1500ms 内不重复弹出，且两者各自独立计时。
17. 登录与缘分墙 `loading` 当前默认最小时长为 600ms（可按体验调整）。
18. 登录手机号仅校验 11 位数字，不限制号段。
19. 缘分墙资料从本地 mock 池按“同日稳定选择”策略取 1 条（mock 池最小 3 条）。
20. mock 池文件路径固定为 `panda-mobile/mock/affinity-profiles.mock.json`。
21. 会员入口以灰态呈现，不可点击。
22. 非 MVP 聊天组件保留文件，但不接入 MVP 主流程。
23. `login_submit_blocked` 在首次进入登录页的初始非法态不触发。

## 4. D5 P0 决策补充（2026-03-11）

1. D5 继续采用 A 路线的“最小扩展”：仅补齐缘分墙动作闭环，不引入真实后端。
2. 缘分墙新增两个动作按钮：`再等等`、`感兴趣`。
3. 同一自然日仅展示 1 个推荐对象；点击动作后不刷新下一张。
4. 日期边界按设备本地自然日（00:00）计算。
5. 当天动作点击即锁定，不可改选（`再等等` 与 `感兴趣` 互斥）。
6. 点击 `再等等`：记录当天动作，不刷新卡片，下一天再推荐。
7. 点击 `感兴趣`：执行固定 mock 匹配规则（`profileId=="p_001"` 为 `matched=true`）。
8. `matched=true` 时跳转消息栏占位页；`matched=false` 时 toast：`今日匹配已用完，请明日再来`。
9. 消息页本轮仅占位，不实现真实聊天发送与会话列表。
10. 从消息栏占位页返回时，按当前 `dateKey` 回到缘分墙应展示卡片（立即重算）。
11. 本轮先更新文档基线，再进入代码实现。

## 5. D5 P0.2 澄清补充（2026-03-11）

1. 同一自然日内重新登录后，仍展示同一个推荐对象。
2. App 进程被杀后，动作锁定不保留（可重新选择动作）。
3. 动作锁定后再次点击为静默忽略（不弹 toast）。
4. `再等等` 文案在 MVP 固定为：`已为你安排明日推荐`。
5. 日期边界保持设备本地自然日，并在本地时间 00:00 自动重置当日状态。
6. `tech-stack.md` 中“待锁定项”推迟到 D5 功能完成后再统一锁定。

## 6. D5 P0.3 澄清补充（2026-03-12）

1. 同日稳定推荐键固定为 `stable_pick_unseen(userId, dateKey, cycleHistory)`。
2. 页面停留中跨越 00:00 时立即重置并换卡，不等待下次交互或 `onShow`。
3. `matched=true` 的会话标识固定按 `conversationId = profileId + "_" + dateKey` 生成。

## 7. D5 P0.4 澄清补充（2026-03-12）

1. 推荐策略升级为“用户历史去重”：同一用户历史已推荐对象后续不再推荐。
2. 00:00 即时重置后应切换到“历史未推荐对象”，而非仅保证与前一日不同。
3. 动作输入示例中的 `userId` 统一为 `mock_user_${phone}` 风格。

## 8. D5 P0.5 澄清补充（2026-03-12）

1. 未推荐池耗尽时，不在当日清空 `cycleHistory`；下一天 00:00 重置 `cycleHistory` 并重新循环推荐。
2. `lifetimeHistory` 仅在“卸载/清除应用数据”时清空。

## 9. D5 P0.6 澄清补充（2026-03-12）

1. 采用双历史模型：`cycleHistory`（参与推荐去重）与 `lifetimeHistory`（长期留痕）。
2. `stable_pick_unseen` 只使用 `cycleHistory` 做“当前循环不重复”约束。

## 10. D5 P0.7 澄清补充（2026-03-12）

1. `stable_pick_unseen` 只要求行为满足产品约束，具体算法实现可自定。
2. `cycleHistory` 与 `lifetimeHistory` 均按 `userId` 维度本地持久化。
3. 消息占位页在 MVP 阶段不处理跨 00:00 的特殊逻辑。
4. 次日 00:00 进入新循环后，首个推荐对象优先不与上一自然日最后展示对象相同（无可选候选可降级重复）。
5. 非 MVP 聊天组件保留文件，但保持未接入状态。

## 11. D5 P0.8 澄清补充（2026-03-12）

1. “上一自然日最后展示对象”按“系统最后一次成功分配给该用户的推荐对象”定义（即使用户当时不在缘分墙页）。
2. 若新循环无法满足“首卡 != 上一日末卡”（无可选候选），允许降级为重复上一日末卡。
3. 消息占位页保持“跨 00:00 不做特殊处理”，但返回缘分墙时需按当前 `dateKey` 立即重算卡片。
4. 非 MVP 聊天组件“未接入”定义为：主流程页面禁止 import，主流程路由禁止跳转到非 MVP 聊天路由。
5. `loading=600ms` 维持“默认可调”口径，不作为固定验收阈值。
6. `login_submit_blocked` 在首次进入页面初始非法态不触发。
7. 历史条目中涉及本节主题的表述已统一为本节口径。
8. MVP 阶段 `nav_back_to_affinity_active` 仅保留基础导航字段，不新增重算结果字段，也不要求重算完成后触发。
9. 消息占位页可将系统返回视为空态的“下一步动作”，MVP 不强制新增显式按钮/链接。
